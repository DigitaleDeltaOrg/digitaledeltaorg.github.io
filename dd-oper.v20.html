Digital Delta - Operational API

Introduction

The Digital Delta is a distributed network in which various data sources exchange information using a standard web service interface. The architecture and additional description of the Digital Delta can be found on ... (TODO refer to digitaldelta.org once it is updated - end of August). The present document contains the specification of the Digital Delta web service API. It uses the RAML standard as the definition language.

The API specification

This API describes 'DD-OPER', which is a series of additional endpoints to the 'DD API'. The structure of the /timeseries response of DD-OPER API is in line with the /timeseries response defined by DD-API, but will provide a different (sub)set of the information.

/dd-oper/2.0/locations get

The Locations endpoint returns the complete list of locations within the NL.RWS.WM namespace that the Provider offers. There are no filterings options.

get

get /dd-oper/2.0/locations

  • Response

HTTP status code 200

Body

Media type: application/json

Type: object

Properties
  • provider: (object)

    Information regarding the system that is responible for the response. Provides support contact information in case there are issues with response format or content.

    • name: (string)

      Name of the provider (of the service).

    • supportUrl: (string)

      Url to information for the provider (of the service).

    • apiVersion: (string)

      API Version with which the result set was generated. Can be used to parse the response.

    • responseType: (string)

      Name of the type of the result set. Can be used to parse the response.

    • responseTimestamp: (string - minLength: 24)

      Timestamp when this response was created.

      Example:

      2018-07-05T15:11:36.123Z, 2018-07-05T15:11:36.123456Z, 2018-07-05T15:11:36.000Z
  • results: required (array of DD-T.Location)

    Items: Location

    • type: required (Feature)

      Type of GeoJSON object. In the DD-API the only allowed type is "Feature".

    • geometry: required (union of DD-T.Geometry or nil)

      The coordinate(s) of the Location are placed within the "geometry" section of the GeoJSON object. DD-API uses Point and Multi-Point locations, but is not restricted to these two.

    • properties: (object)

      The attributes of the Location object are placed within the properties group of the GeoJSON object.

      • url: (string)

        Url describing the location object. (Not provided in case of /dd-oper requests.)

      • node: (object)

        Specification of the node where the location is stored.

        • id: (string)

          Id of the node.

        • name: required (string)

          The name of the node.

        • description: (string)

          The description of the node.

        • baseUrl: (string)

          The base URL of the node.

      • locationId: (string)

        Id of the location.

      • locationCode: (string)

        Code of the location.

      • locationName: (string)

        Name of the location.

        Example:

        Hoek-van-Holland
      • referenceLevel: (string)

        GeoJSON element. Specifies the reference plane for the altitude coordinates (third). For example "NAP".

      • crs: (object)

        GeoJSON Coordinate Reference System element. According to GeoJSON the CRS always as to be "WGS84", hence the crs element has been removed from the GeoJSON Feature object. However, the DD-API still has the field, to be able to specify that the data is provided in another CRS (although this is discouraged).

        • locationNameSpace: (string)

          Remark: Only used in DD-OPER response.

          Example:

          NL.RWS.WM
        • displayNameDetail: (string)

          Remark: Only used in DD-OPER response.

          Example:

          Hoek-van-Holland
        • displayNameGlobal: (string)

          Remark: Only used in DD-OPER response.

          Example:

          Hoek-van-Holland
        • displayNameSpace: (string)

          Remark: Only used in DD-OPER response.

          Example:

          NL.RWS.WM
        • parentName: (string)

          Remark: Only used in DD-OPER response.

          Example:

          Hoek-van-Holland

    Example:

    /dd-oper/2.0/locations/{locationName}/quantities/{quantityName}/timeseries get

    get

    Retrieve resource

    get /dd-oper/2.0/locations/{locationName}/quantities/{quantityName}/timeseries

    Retrieve resource

    • Request
    • Response

    URI Parameters

    • locationName: required (string)

      Example:

      Hoek-van-Holland
    • quantityName: required (string)

      A name referencing a category of time series. The quantity names are predefined. For example 'WaterLevel', 'WaterTemperature', 'WaveHeight'. Remark: Only used in DD-OPER

      Example:

      WaterLevel

    Query Parameters

    • startTime: required (datetime)

      Start date/time of the time series, or the Start date/time of the interval for an individual value in the series. Uses the UCT (Zulu) notation.

      Example:

      2016-01-01T12:45:00Z
    • endTime: (datetime)

      The End date/time of the time series, or the End date/time of the interval for an individual value in the series. Uses the UCT (Zulu) notation. The endtime is exlusive, hence the requested period ends just before the given time. 2018-01-01T00:00:00Z - 2018-01-01T00:00:10Z specifies a 10 minute period starting at midnight and ending at 2018-01-01T00:09:59.99999Z

      Example:

      2018-01-01T00:00:00Z
    • intervalLength: (string - default: 10min)

      Interval length of the sequence.

    • aspectSet: (string - default: standard)

      Name of an AspectSet. If implementing systems require an aspect-filter in the request-url and it is omitted, "default" is taken.

      Example:

      MinimumMaximumAverage
    • process: required (string - default: measurement)

      The name of the mechanism to collect/derive the time series value. When accessing RWS' DD-API, providing a processName as query parameter is required. The process names are pre-defined (see examples). Extending the processname with a suffix, referencing a more detailed mechanism like a specific calculation model, is not allowed. Other implementions dot need to implement this query-parameter.

      Example:

      measurement, forecast, astronomical, advise
